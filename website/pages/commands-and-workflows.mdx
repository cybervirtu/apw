# Commands & Workflows

This is the operator layer for APW.

Use it once the repo exists and you want the practical answer to:

- which command should I use?
- when should I use it?
- which agent should I pair with it?
- when do I hand work back to the orchestrator?

## The practical invocation pattern

```text
@agent /workflow task description
```

Examples:

```text
@orchestrator /status summarize the current state and next best action
@backend-specialist /create implement password reset API from TODO.md
@qa-automation-engineer /test verify checkout integration flow
```

## Quick chooser

| Situation | Workflow |
| --- | --- |
| returning after a break | `/status` |
| shaping an idea before implementation | `/brainstorm` |
| building something new | `/create` |
| improving existing code | `/enhance` |
| fixing a bug | `/debug` |
| verifying quality or completion | `/test` |
| planning frontend structure | `/design` |
| polishing UI quality | `/ui-ux-pro-max` |
| preparing local review | `/preview` |
| preparing release or deployment | `/deploy` |
| decomposing a large cross-cutting task | `/orchestrate` |

## The APW execution rule

<Steps>
### Read context first

Start from `AGENTS.md`, then read the minimum required `.gsd` context for the task.

### Do scoped work

Use the workflow that matches the job instead of issuing a vague prompt.

### Keep evidence bounded

Execution agents may append bounded evidence to `.gsd/JOURNAL.md`.

### Let the orchestrator synchronize official state

Execution agents should not freely rewrite `STATE.md`, `ROADMAP.md`, `TODO.md`, or `DECISIONS.md`.
</Steps>
