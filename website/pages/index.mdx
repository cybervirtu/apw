# APW Docs Portal

This portal is the visual front door for APW.

APW helps humans and AI agents build software without losing track of project state, governance, or execution context.

<Callout type="info" emoji="🧭">
This homepage is a guided entry page. The canonical APW rules still live in the repo-root governance files and the source docs under `docs/`.
</Callout>

<Callout type="info" emoji="🪜">
APW now exposes its docs in three layers: Start Now, Use APW Better, and Deep Reference.
</Callout>

<Callout type="info" emoji="🛠️">
For a brand-new repo, think `APW: Create Project` first. Terminal fallback: `/path/to/apw/apw new MyProject --profile base --stack base`.
</Callout>

<Callout type="info" emoji="⚙️">
If you still need to install APW locally, start with the Installation Guide first.
</Callout>

<Callout type="info" emoji="📋">
If you just want the important commands fast, keep the repo's `COMMAND_CHEATSHEET.md` nearby.
</Callout>

<Callout type="info" emoji="🪄">
APW is not terminal-only. The preferred path is chat-first request, then IDE action, then terminal fallback.
</Callout>

<Callout type="info" emoji="🌱">
For beginners, the first APW actions to remember are: Create Project, Initialize Project State, and First Run.
</Callout>

<Callout type="info" emoji="🧭">
For orientation and safe switching, think in these actions: Show Context, List Projects, Switch To Project, Switch To Framework, and Switch To Parent.
</Callout>

<Callout type="info" emoji="📍">
Normal project work happens in the downstream project root. APW root is mainly for framework maintenance, and the workspace parent folder is mainly an organizer.
</Callout>

<Callout type="info" emoji="🔀">
If you lose track of the right folder, think Show Context, List Projects, or Switch To ... first. Terminal fallback: `apw context`, `apw list-projects`, or `apw switch ...`.
</Callout>

<Callout type="info" emoji="🧠">
If you use `/brainstorm`, do not assume the chat became project memory automatically. Save useful outcomes deliberately.
</Callout>

<Callout type="info" emoji="📝">
Across the core workflows, APW keeps the same persistence rule: bounded evidence first, deliberate canonical sync second.
</Callout>

<Callout type="info" emoji="🔄">
If APW itself evolves later, think Preview Upgrade first. Terminal fallback: `apw upgrade-project <name-or-path> --dry-run`.
</Callout>

<Callout type="info" emoji="🛡️">
For upgrade work, think in this order: Preview Upgrade, Upgrade Project, then Validate After Upgrade.
</Callout>

## Who APW is for

APW is useful for:

- beginners who want a safer way to start AI-assisted projects
- builders who want repeatable bootstrap, validation, and CI guardrails
- teams who need shared project memory instead of chat-by-chat drift
- maintainers who want one framework that works with Codex, Antigravity, and related tooling

## The APW model in one view

| Part | What it does |
| --- | --- |
| GSD governs | protects lifecycle, planning, verification, and canonical project memory in `.gsd/` |
| AGK executes | provides specialist agents, workflows, skills, and execution support in `.agent/` |
| Orchestrator synchronizes | keeps official state changes controlled instead of letting execution notes become the project plan |

## How APW works

<Steps>
### Bootstrap the workspace

Create the repo structure from anywhere in the workspace and choose the right APW profile for the project.

### Capture the first real project state

Run the guided initializer so APW can turn the idea into the first drafts of the core `.gsd` files.

### Execute scoped work

Use the right agent and workflow instead of issuing vague prompts.

### Record bounded evidence

Execution work may add evidence to `.gsd/JOURNAL.md` without rewriting the whole project memory.

### Synchronize official state

Let the orchestrator update canonical state when `STATE.md`, `ROADMAP.md`, `TODO.md`, or `DECISIONS.md` need to change.

### Validate and continue

Use validation and CI to keep the repo aligned as work continues.
</Steps>
