# What Is APW?

APW stands for **Agentic Project Workspace**.

It is a framework for running software projects with humans and AI agents in a way that stays understandable over time.

This page is a beginner-friendly explanation. The canonical APW contract still lives in the repo-root governance files and the deeper docs under `docs/`.

In plain English:

- APW gives the project a memory
- APW gives agents a place to work
- APW gives the team a way to keep that work aligned over time

AI tools move fast, but they drift effortlessly. APW prevents this by separating **Memory** (what is true), **Execution** (how we work), and **Enforcement** (how we stay compliant).

For a deeper dive into the "Why", see [APW for Beginners](file:///Users/My/Documents/Jewelminds/apw/docs/APW_FOR_BEGINNERS.md).

## The APW model

APW is built on a "Dual-Engine" model where **GSD** manages the brain (planning/state) and **AGK** provides the muscle (implementation/skills).

| Part | Layer | Responsibility |
| --- | --- | --- |
| **GSD** | `.gsd/` | Project memory, SPEC, ROADMAP, STATE |
| **AGK** | `.agent/` | Execution rules, workflows, specialists |

For the full architectural breakdown, read [How APW Works](file:///Users/My/Documents/Jewelminds/apw/docs/HOW_APW_WORKS.md).

## The team model in plain language

<Steps>
### GSD governs

GSD owns lifecycle, verification, and canonical project memory.

### AGK executes

AGK provides the specialist execution help that builds, debugs, tests, and improves the project.

### The orchestrator synchronizes

The orchestrator reads execution results and updates canonical state when the official summary needs to change.
</Steps>

## What APW helps you do

APW helps you:

- start a project with structure instead of prompt chaos
- keep AI sessions aligned with shared project memory
- separate implementation evidence from official status
- hand work between sessions or teammates more safely
- validate and enforce the repo structure over time

## Who APW is for

APW is useful for:

- beginners using AI coding tools for the first time
- solo builders who want stronger project memory
- product teams working across multiple AI sessions
- maintainers who need one framework instead of tool-specific forks
