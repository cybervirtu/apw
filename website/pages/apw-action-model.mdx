# APW Action Model

Use this page when you want the simple APW interaction rule beyond raw terminal commands.

<Callout type="info" emoji="🪄">
APW's preferred hierarchy is: chat-first request, then IDE action, then terminal fallback.
</Callout>

Canonical source doc: `docs/APW_ACTION_MODEL.md`

## The short version

APW uses one user-facing action catalog for common tasks such as:

- Create Project
- First Run
- Show Context
- List Projects
- Switch To Project
- Switch To Framework
- Switch To Parent
- Initialize Project State
- Upgrade Project

These action names are the preferred chat and future IDE labels.

Underneath, they still map to the existing APW command layer.

For project creation, the default destination is workspace-aware: from APW root, the new repo is created beside <code>apw</code>; from the workspace parent, it is created in the current folder; from a downstream project, it is created as a sibling repo in the same workspace parent. Chat-first project creation should show that resolved destination before anything is created.

## The first beginner action layer

If you only remember the first APW actions, remember these:

- `APW: Create Project`
- `APW: Initialize Project State`
- `APW: First Run`

These are the three actions beginners should think about before they think about raw terminal commands.

## The context and switching action layer

Once the project exists, the next APW actions to remember are:

- `APW: Show Context`
- `APW: List Projects`
- `APW: Switch To Project`
- `APW: Switch To Framework`
- `APW: Switch To Parent`

These are the preferred chat and future IDE labels for orientation and safe switching.

## The upgrade action layer

When APW itself changes and a downstream repo needs the newer APW-managed layer, the preferred action sequence is:

- `APW: Preview Upgrade`
- `APW: Upgrade Project`
- `APW: Validate After Upgrade`

This keeps the upgrade flow preview-first and makes the protection boundary visible.

## Why this matters

- beginners do not need to think in terminal flags first
- IDE-native behavior can stay aligned with the current APW engine
- terminal commands remain available for fallback, power users, and automation

Read the repo's `docs/APW_ACTION_MODEL.md` for the canonical action catalog and command mapping.
