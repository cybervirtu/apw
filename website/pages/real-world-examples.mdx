# Real-world Examples

This page answers a simple question:

"Can APW help with the kind of project I actually want to build?"

The answer is usually yes, but the profile, stack direction, and first workflow steps will vary by project shape.

## Example map

| Project idea | Likely profile | Likely stack direction | Likely first move |
| --- | --- | --- | --- |
| Personal portfolio or landing page | `minimal` or `base` | static site or lightweight web frontend | bootstrap, validate, then `/design` or `/create` |
| SaaS dashboard or admin tool | `base` | frontend + backend API + database | initialize `.gsd`, then `/status`, `/design`, or `/create` |
| AI support chatbot | `base` or `advanced` | app/API layer plus focused AI feature layer | `/brainstorm`, then `/create` or `/orchestrate` |
| Internal reporting tool | `base` | dashboard or internal web app + backend + database | `/create` around the first useful reporting slice |
| Multi-service platform | `advanced` | monorepo or multi-service setup | `/orchestrate` early, then specialist execution |

## Sample project paths

- **Personal portfolio:** choose `minimal` or `base`, bootstrap cleanly, and start with the first page or layout slice.
- **SaaS dashboard:** choose `base`, define the first usable flow, initialize `.gsd`, and use `/design` or `/create` for the first milestone.
- **AI support chatbot:** choose `base` or `advanced`, keep the first support scope narrow, and use orchestrator help early if the work cuts across several specialties.
- **Internal operations tool:** choose `base`, define the first business problem to improve, and keep the first reporting or workflow slice intentionally small.
- **Monorepo platform:** choose `advanced`, confirm the project really needs multiple services, and decompose the work before implementation starts.
- **Backend service:** choose `base`, define the first endpoints or integrations, and keep the service narrative visible in `.gsd`.

## The APW pattern across all examples

No matter which project you choose, APW keeps the same broad flow:

1. clarify the idea
2. choose the profile and likely stack direction
3. bootstrap and validate
4. initialize `.gsd`
5. start scoped work
6. log bounded evidence
7. sync canonical state deliberately
